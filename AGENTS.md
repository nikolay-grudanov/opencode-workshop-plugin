## What this project is

This is **Kolya's maintained fork** of the npm-only `@raindrop-ai/opencode-plugin` (the closed-source plugin used by Raindrop Workshop to stream OpenCode sessions/spans into the local Workshop daemon on `http://localhost:5899`).

The upstream plugin source is **not on GitHub** — it ships only as an npm tarball (`@raindrop-ai/opencode-plugin@0.0.18`, 41 KB, MIT). We extracted the tarball, initialized a public git repo, and now maintain our own branch.

**Public package name:** `@grudanov-nikolay/opencode-workshop-plugin` (drop-in replacement for `@raindrop-ai/opencode-plugin`).

**Public repo:** https://github.com/nikolay-grudanov/opencode-workshop-plugin

**Why the fork exists:**

1. **Upstream bug [anomalyco/opencode#21149](https://github.com/anomalyco/opencode/issues/21149)** — for MCP tool calls, OpenCode passes the raw `CallToolResult` (`{content: [{type, text}]}`) to the `tool.execute.after` hook, but the plugin's contract expects `{title, output, metadata}`. The plugin's `if (!("output" in result)) throw` crashes every MCP tool call with `tool.execute.after: result.output is required`. Upstream PR #21150 was **auto-closed on 2026-05-15** (no positive reactions) — fix will never come from upstream. We patch it.
2. **Env var precedence footgun** — `RAINDROP_LOCAL_WORKSHOP_URL` env var unconditionally overrides `raindrop.json` `local_workshop_url`, even when set to a stale or non-local value. We've been bitten by this on 2026-06-30 (lost spans for 2 hours).
3. **Workshop event attribution** — Kolya needs per-project `eventName` to split traces in Workshop UI (`oc-kd`, `oc-diagram`, `oc-notraces` wrapper pattern). Upstream has this partially, but needs hardening for multi-project setups.

## Working conventions in this repo

### PLAN.md is the single source of truth

- All work is organised as **Features**.
- Each Feature has its own section in [`ai-docs/PLAN.md`](ai-docs/PLAN.md).
- Within a Feature, work items are listed as markdown checkboxes:
  - `- [ ] todo text` — pending
  - `- [x] todo text` — done
- **Always update PLAN.md in the same commit as the code change.** If you finish a todo, mark it `[x]` AND commit. Don't let the plan drift from the code.
- New features go at the **top** of the file (newest first), with the next available F-number.

### Do not touch

- ❌ `dist/index.js` and `dist/index.cjs` — generated by tsup from `src/index.ts` (once we add the build). Hand-editing them defeats the build chain. (In v0.1.0-kolya.1 we ship hand-edited dist/ because we don't have src/index.ts yet — see Feature F-001.)
- ❌ `node_modules/`, `dist/` — gitignored.
- ❌ `package-lock.json` if the project is pnpm-based; otherwise OK to commit.

### Build & test (planned, not yet wired)

```bash
# Once F-002 is done:
pnpm install
pnpm build           # tsup → dist/index.{js,cjs,d.ts}
pnpm test            # tests live in tests/ (currently empty)
```

### Publishing

```bash
# To npm under @grudanov-nikolay scope:
pnpm publish --access public --tag kolya
```

### Git workflow

- One commit per logical change. Conventional commits: `fix:`, `feat:`, `chore:`, `docs:`.
- Reference the Feature ID in the commit body, e.g. `Closes F-001`.
- Never push to main without Kolya's explicit "push" or "merge" command.
